## 贯穿洞察（M6 的方法论沉淀）

### 1. 安全门是一个"门的谱系",按可确定性分层
- **确定性检测器**(M6-a 密钥 / M6-b 敏感面):regex/路径分类,纯函数**可 TDD、可硬门**——这是可靠骨架。
- **非确定 LLM 评审**(M6-d STRIDE):语义级威胁(注入/越权/失败副作用)规则覆盖不到,需"读懂意图"。但 LLM 非确定 → **不单独硬阻断**(漏报风险),产 findings 交确定性映射器,**advisory + 高危人在环**,与确定门**互补不替代**。
- **沉淀**:把非确定性**包在纯函数边界外**——agent 产结构化输出(findings/verdict),纯 mapper 做确定动作。这是把不可靠 agent 纳入可靠 harness 的通用配方(同既有 review→verdict→mustFix)。

### 2. 门的"动作"分两类,对应两种问题
- **must-fix**(agent 自修):M6-a 密钥——硬编码凭证不该存在,移除/参数化是 agent 能做且该做的(像 lint)。改"内容错了"。
- **escalate-hold**(人签):M6-b 敏感面 / M6-d 高危——改动**合法、必需**,但触及鉴权/CI/基础设施或确有可利用漏洞 → 不能自动合,人裁决。改"谁批准"。复用 held/handoff,不丢工作(非 escalated)。
- 这正是红线7「人类门禁不可绕过」的机制化:机器能判的机器判,机器判不了的(谁该批准)交人。

### 3. "不影响已实现功能"= 可工程化的三重保证(贯穿 a/b/d)
每个切片都用同一手法兑现用户硬约束,非口头承诺:
1. **向后兼容注入**(默认 no-op):新门作可选注入(secretScan/sensitiveCheck/可调用安全评审)→ 既有调用方/测试**零行为变化**(222→232→240→249,每步既有测试零改动通过)。
2. **范围限定**:只扫/判**本次改动**(diff/feature 路径),不回溯全树。
3. **真实零误报/零回归验证**:fincards 产品全量零误报;既有全量测试零改动通过。

### 4. 阈值不臆造(NFR)
NFR 上游"方向占全 + 阈值不臆造":未做长程任务测试 → 性能/成本/长程可靠性 SLO 一律标"待长程标定"。与 M4+「待埋点」同源:宁可标待定,不伪造没测过的数字(红线1)。**已达标维度**(可观测/安全/并发/可移植)其实已被既有门满足——NFR 不是立一堆新门,而是给方向占名 + 标清缺哪几维数据。

### 5. 失败驱动 + 意外收获
- M6-a 被 RESUME §6 真实 **PAT 泄露事件**拉出来(红线3:约束被真实失败拉出,不凭空加)。
- **意外**:`*secret*` gitignore 子串模式差点把 secret-**扫描器**代码静默丢出 git(安全门代码没进库=最危险的"影响已实现功能")——子串-anywhere 忽略模式是潜伏 footgun;收窄为精确密钥文件模式,且新门本身已是提交期真保护。
