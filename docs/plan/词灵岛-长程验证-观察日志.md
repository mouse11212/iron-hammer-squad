## 1. 验证方法（怎么"监控开发链路"）

- **从最小垂直切片起步**（§13 / PRD §9）：首切片取三年级最小闭环，先跑通"登录选年级→进一关→听音选词→拿金币→15 分钟落幕"骨架，再按外循环扩。
- **每个 US 过内循环**（driver）：测试 Agent 先写用例(RED) → 开发 Agent TDD(GREEN) → 门禁 → 评审 → squash。我观察每一步质量与是否守纪律。
- **逐 US 记录**：在 §3 内循环台账填一行（规约对齐？测试先行？门禁有效？评审抓到问题？drift？成本/耗时？）。
- **架构符合度**（§4）：对照构思 v4 三层模型 + 8 军规 + 7 红线 + 角色路由逐项核对。
- **问题→优化闭环**（§5/§6）：发现的问题进台账；能固化的产出对 harness 的 diff。

## 2. 验证目标 / 成功判据（据实，不预设阈值）

- **内循环可用且质量好**：能把一个 US 从规约→测试→实现→门禁→评审→可合并跑完，产出可运行且测试有效的代码；门禁能真实拦住坏代码；评审能抓到测试看不到的问题。
- **外循环可用**：能把 PRD 拆成 epic→US→task，按依赖排程，逐 US 推进、bug 能回流修复。
- **过程守架构**：角色不混同、测试先行不被弱化、人类门禁不被绕过、从窄到宽、提交/分支守军规。
- **据实**：NFR（每 US 成本/耗时、长程可靠性）、drift 信号一律记真实测得值，测不到的标 `insufficient-data`，不臆造。
