# fincards 需求澄清 v1（产品/澄清 Agent 产出）

> 由 M2-B 用 gstack office-hours(YC 逼问)方法产出,偿还 M0 延后的需求澄清债(V4 §10 需求澄清 DoD)。
> 性质:上游产物,喂给未来 OpenSpec 规约(V4 §3.3 需求澄清 → … → 规约),非 capability 规约本身。
> 日期 2026-06-17。

## 产品概念

fincards:每日自动聚合**权威财经新闻**,为**个人投资者/交易者**生成"自动提重点"的速读总结页(Top 卡片 + 简略文字),点卡片跳原文深读。

## 目标用户画像(demand reality)

- **个人投资者/交易者**(散户到半专业):股/期/加密等。
- 特征:每日需快速掌握市场要闻;时间有限;自己刷多个 App/网站手动筛(status quo)。

## 核心痛点(desperate specificity)

**信息过载——新闻太多,缺一个"自动提重点"的结构化速读。** 现状靠人工在多源间翻找,累、易漏重要消息。fincards 的价值=把"提重点"自动化。

## 业务场景

- 每日(早晨/盘前)打开总结页速览市场要闻。
- 对感兴趣的卡片点链接跳原文深读(合规导流)。

## 范围

**v1 已有(M0/M1)**:单源(Bloomberg 官方 RSS)→ 当天新闻 → 卡片页 + 确定性 gate + 变异门。

**future-fit 优先级(用户拍板,注意顺序)**:
1. **多源聚合**(下一步最想要)——接入更多权威财经源。
2. Top10/20 排序选取(按热度/影响"提重点")。
3. LLM 中文摘要/提炼。
4. 定时自动每日生成(事件/定时触发)。

> 张力提示:痛点是"提重点",但用户把"多源聚合"排在"排序"之前——意味着**先把源做全,再做提重点**。多源会加剧信息过载,因此 (1) 完成后 (2) 排序的紧迫性会上升。

**out-of-scope(v1 明确不做)**:账号/个性化、实时推送、正文镜像(合规——只导流原文)、交易执行/投顾建议。

## 风险与边界(§12 规格模糊上界)

- **"提重点"的排序口径需显式定义**:热度/影响范围如何量化?在做排序(优先级 2)前必须把"重要"变成可判定规则,否则正确性出 sensor 职责范围(Ashby)。
- **合规**:仅用官方 RSS 公开字段、导流原文、不镜像正文(已固化)。
- **多源异构**:不同源 RSS 字段/时区/质量不一,解析器需适配(复用 M0 的薄 IO + 纯解析分层)。

## 对下游的指示

- M2-A(多角色跑真实功能)的首选增量 = **多源聚合**(对齐用户优先级 1),而非排序。
- 排序(优先级 2)在多源完成后单列里程碑,且需先定"重要性"口径。
