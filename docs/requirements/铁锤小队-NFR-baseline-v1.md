# 铁锤小队 · NFR 基线 v1（harness 引擎层）

> 用途：M6（NFR 门 + 安全门）的 **NFR 上游 SoT**——M6-c「NFR 派生测试门」从这里挑**已标定**维度派生测试入门禁。
> 定位：北极星级人读文档（非机器 SoT）。回填 V4 §8 的「[待完善：SLO 值]」。最后更新 2026-06-25。

## 0. 范围与原则

- **范围**：本文件定义 **harness 引擎（`pipeline/`）自身**的 NFR 基线。**产品级 NFR**（如 fincards 的渲染时延/抓取超时）留各产品 `design.md` 实例化，不在此。
- **NFR 两条硬要求**（V4 §8）：(a) NFR 必须**派生自己的测试用例**并进入门禁；(b) NFR 是**横切、持续验证**项，非一次性。
- **阈值哲学（关键约束）**：
  - **方向要全**——该有的质量维度都列到（红线3 从窄到宽的"窄起点"=先把方向占全）。
  - **阈值不臆造**——当前工程**尚未做长程任务测试**，凡需长程数据才能定的 SLO（性能/成本/长程可靠性）一律标 **「待长程标定」**，**不编造没测过的数字**（红线1）。这与 M4+ 的「待埋点」同源:宁可标"待定"，不伪造。
  - **已达标的如实标 ✅**——已被既有确定性门满足的维度（可观测/安全/并发等）标为"已达标基线"，避免重复造门。
- **阈值状态三态**：`✅ 已达标`（既有门满足）/ `保守基线`（有阈值但偏松，按真实数据收紧）/ `待长程标定`（方向已占，SLO 值留长程测试后回填）。

## 2. 各维度详解（含 M6-c 测试派生方向）

> 标注每维：**直接派生测试**（M6-c 可立即做门）/ **待标定后派生**（需长程数据先定阈值）/ **已有门覆盖**（无需新增）。

### 1. 可观测性 — 已有门覆盖
- 测试派生：四指标采集纯函数已穷尽单测；traceId 全链回放有 e2e。**无需新增 NFR 门**。

### 2. 安全 — 已有门覆盖
- 测试派生：M6-a/b 已是门（密钥扫描入 green、敏感审批入集成）。**无需新增**；后续 M6-d/e 扩面。

### 3. 并发安全 — 已有门覆盖
- 测试派生：M5-A 已有"N 进程抢 M 条零双领"压测。**无需新增**。

### 4. 可移植/可恢复 — 已有门覆盖
- 测试派生：worktree 隔离/回收、main-HEAD-不变、状态恢复均有测。**无需新增**。

### 5. 可维护性 — 保守基线，可直接派生
- 方向：纯逻辑变异门 ≥ 90（driver 现有 break 90）。
- M6-c 可做：把"变异门 ≥ 阈值"显式纳入 NFR 门表述；**补 metrics 包级变异门**（现靠穷尽精确断言兜底，是已知零头）。阈值 90 为保守基线，按真实弱测数据收紧。

### 6. 可靠性 — 机制就位，阈值待长程标定
- 方向：崩溃可恢复、瞬时重试、终态确定。机制均已落地（M3/重试/止损/动态 squash）。
- **待长程标定**：连续无人值守跑 N 个 US 的**成功率**、**平均无人工干预间隔**——需长程任务测试产生真实数据才能定 SLO。标定前不设硬阈值（不臆造）。
- M6-c 可做（标定后）：从 runs-ledger 派生"滚动窗口升级率 ≤ X"门。

### 7. 性能/时延 — 机制就位，SLO 待长程标定
- 方向：单 US 墙钟不无限（maxFixRounds 止损）；gate/phase 时长已埋点（durationMs）。
- **待长程标定**：单 US 端到端墙钟的 p50/p95 上限——现仅 2 个真实样本（relativeTime 243s、canonicalizeUrl 856s），**样本不足以定 SLO**。需长程批量跑积累分布。
- M6-c 可做（标定后）：从 events.jsonl/history 派生"单 US 墙钟 ≤ X 分钟"软门。

### 8. 成本 — 可观测，上限待长程标定
- 方向：单 US 成本已埋点（ledger costUsd）；超限应升级/止损。
- **待长程标定**：单 US 成本上限——需长程样本看真实成本分布。现样本不足。
- M6-c 可做（标定后）：从 ledger 派生"单 US 成本 ≤ $X 否则升级"门。

## 3. 刻意排除（YAGNI，红线3）

| 维度 | 排除理由 |
|---|---|
| usability / a11y | harness 是 dev-facing CLI/编排，非终端用户 UI。**留产品 UI 阶段**在各产品 design.md 定（如 fincards 的卡片可读性）。 |
| 合规 | 当前无监管/合规场景。出现具体合规要求时再立。 |
| 可伸缩（scalability） | 当前规模下「并发安全（#3）」已覆盖所需；真正的水平伸缩（多机/分布式）超出当前本地非云常驻定位（D9）。 |

## 4. 标定计划（待长程测试后回填）

**触发条件**：当工程开始**长程任务测试**（连续无人值守跑批量 US）后：
1. 从 `runs-ledger.jsonl` / `events.jsonl` / `history.jsonl` 积累真实分布。
2. 对 #6/#7/#8（可靠性/性能/成本）按 p50/p95 定**保守初始 SLO**（取实测分布的宽松分位，留余量）。
3. 回填本表"待长程标定" → 具体值，并交 **M6-c** 把已标定维度派生为测试入门禁。
4. SLO 随真实数据**持续收紧**（NFR 是横切持续验证项，非一次性 / V4 §8）。

> **纪律**：标定值必须来自真实测量（红线1）；标定前 M6-c 只对"已达标/保守基线"维度（#5 可维护性）派生门，不对"待标定"维度凭空设阈值。
