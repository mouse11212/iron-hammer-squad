铁锤小队A HARNESS-LEVEL AI SDLC PIPELINE

给定需求，最大限度地用 AI 团队完整跑通从需求到交付的 SDLC。追求的不是「看起来全自动」，而是 在已知可靠性边界内，可观测、可度量、可问责地交付。核心哲学：人定目标、人判质量、人担责；AI 编排执行。 对抗的根本问题——单步误差在长链路中的复利累积（agent drift），用纪律化的 harness 把误差拦在放大之前。

工程定位 · §1

三层楼模型：各司其职，互不冲突

范围可画满整条 SDLC（① 层 scope），但 harness（② 层）从窄到宽逐步加固——Start narrow, measure, then expand。 过度约束是已知失效模式。

③

Compound / Steering — 越用越强

每次失败 → 人工诊断根因 → 固化进下层 → 该类错误结构性不可重现。复盘的产出物 = 对 rules / lint / gate / skill 的具体 diff，不是一份报告。这是区别于 VibeCoding（每次错了重新提示）的定义性机制。

/retro/learn反馈固化
②

Harness — 单次执行可靠

流水线每一步在已知边界内单次执行可靠。控制一切归两类、缺一不可：Guides（前馈，行动前引导）+ Sensors（反馈，行动后观测）+ Quality Gates（强制门）。实施顺序：约束 → 反馈循环 → 质量门。

GuidesSensorsQuality Gates
①

Loop Engineering — 无人值守反复跑 SDLC

把「那个反复给 agent 敲提示的人」替换为一套替你做这件事的系统。外循环（工程级 SDLC）+ 内循环（单 US，最小可靠单元）。六部件：automations（触发层）/ worktrees / skills / connectors / sub-agents / state·memory。

一份评审产物，喂两个时点：实现前测试 + 实现后验收

确定性功能门只验「正确性」，验不出「目的是否达成」（issue#12）；UI 更无确定性功能门（issue#9）。 解法不是再加一道盲门，而是让 design-soundness 评审在实现前产出结构化 DesignFindings——「可测」半边喂确定性测试（杠杆1），「不可测」半边喂实现后验收（杠杆2）。同一份评审，两个时点，零重复。

治理 · 两套规则的分层

7 红线（原则层） / 8 军规（执行层）

两层独立编号、互补不冲突；万一抵触，以红线为准并升级人类调整军规。引用务必带前缀，严禁裸用数字。

核心禁止事项 · 高层红线

原则层 — 贯穿全 SDLC，定「什么不可逾越」
  • 禁止臆造：不编造无来源的数字/事实/决策
  • 禁止跳过 KB：AI 领域决策前必先查知识库
  • 禁止一次性拉满：harness 从窄到宽
  • 禁止角色混同：写测试 ≠ 写实现
  • 禁止弱化测试：变更测试需独立记录与门禁
  • 禁止替人拍板：未知/缺信息 → 阻塞升级
  • 人类门禁不可绕过：范围/架构/安全/部署须签字

驭手 8 军规 · Git 执行护栏

执行层 — 定「在 Git 上怎么落地」
  • 智能体只提议——合并是 HITL 决策点
  • 隔离一切——绝不直接动 main，AI PR 加严审查
  • Worktree 并行——端口/DB/密钥也各自独立
  • 短命快合——几小时到几天合回主干
  • Feature Flags——半成品安全合入
  • 规范命名 + 提交——提交捕捉「为什么」
  • 严格审查——永远看 diff，讲不清就不发布
  • 集成分支兜底——汇总全绿再合 main
